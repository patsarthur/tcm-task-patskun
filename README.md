## Частина A — MySQL оптимізація

### Що змінив у схемі і чому

Стара схема мала кілька проблем які я виправив:

**1. `created_at` зберігався як текст (`VARCHAR(32)`)**
Це погано тому що MySQL не може нормально порівнювати дати як текст.
Наприклад `'2025-09-17'` > `'2025-09-15'` працює випадково, а не як дата.
Змінив на `DATETIME` — тепер індекс працює коректно.

**2. URL дублювався в кожному рядку**
Якщо мільйон переглядів `/product/42` — рядок `/product/42` зберігається мільйон разів.
Виніс URL в окрему таблицю `pages`, у `page_views` зберігається тільки числовий `page_id`.
`GROUP BY` по числу працює швидше ніж по рядку.

**3. Не було індексів**
Без індексів MySQL читає всю таблицю при кожному запиті.
Додав індекс `idx_date_country_page (created_at, country, page_id)` під запит A1.

**4. Дрібні типи**
- `country VARCHAR(3)` → `CHAR(2)` — країна завжди 2 символи (UA, PL, DE)
- `device VARCHAR(16)` → `ENUM` — фіксований набір значень, займає 1 байт

### Запит A1 — Топ-5 сторінок за унікальними користувачами

Було (повільно, немає індексів, дата як текст):
```sql
SELECT url, COUNT(DISTINCT user_id) AS uniq_users
FROM pv
WHERE created_at >= '2025-09-15'
  AND created_at <= '2025-09-17'
  AND country = 'UA'
GROUP BY url
ORDER BY uniq_users DESC
LIMIT 5;
```

Стало (працює через індекс):
```sql
SELECT
    p.url,
    COUNT(DISTINCT pv.user_id) AS uniq_users
FROM page_views pv
JOIN pages p ON p.id = pv.page_id
WHERE pv.created_at BETWEEN '2025-09-15 00:00:00' AND '2025-09-17 23:59:59'
  AND pv.country = 'UA'
GROUP BY pv.page_id, p.url
ORDER BY uniq_users DESC
LIMIT 5;
```

### Запит A2 — Batch Insert

Вставляти рядки по одному — дуже повільно.
Краще вставляти пачками (батчами) і загортати в транзакцію.

Погано — 1000 окремих запитів:
```sql
INSERT INTO page_views (...) VALUES (...);
INSERT INTO page_views (...) VALUES (...);
-- ... ще 998 разів
```

Добре — два батчі по 500 рядків в одній транзакції:
```sql
START TRANSACTION;

-- Батч 1 (500 рядків)
INSERT INTO page_views (user_id, page_id, country, device, created_at, duration_ms)
VALUES
    (122, 1, 'FR', 'desktop', '2025-10-09 17:41:14', 1715),
    (188, 1, 'FR', 'mobile',  '2025-10-11 03:06:06', 1992),
    -- ... інші 498 рядків
    (179, 3, 'UA', 'desktop', '2025-09-20 19:43:51', 726);

-- Батч 2 (500 рядків)
INSERT INTO page_views (user_id, page_id, country, device, created_at, duration_ms)
VALUES
    (192, 2, 'GB', 'mobile', '2025-09-27 16:55:51', 701),
    (177, 1, 'GB', 'mobile', '2025-09-23 23:53:55', 156),
    -- ... інші 498 рядків
    (110, 2, 'DE', 'mobile', '2025-09-26 12:22:06', 933);

COMMIT;
```

Чому два батчі а не один на 1000 — великий батч з'їдає більше пам'яті,
500 рядків за раз це оптимальний баланс між швидкістю і споживанням ресурсів.

Якщо даних дуже багато і вони є у CSV файлі — є ще швидший варіант `LOAD DATA INFILE`.
Я згенерував тестовий файл `page_views.csv` на 10 000 рядків для демонстрації:

```sql
LOAD DATA INFILE 'D:/OSPanel/userdata/temp/upload/page_views.csv'
INTO TABLE page_views
FIELDS TERMINATED BY ','
LINES TERMINATED BY '\n'
(user_id, page_id, country, device, created_at, duration_ms);
```

## Частина B — PHP ReportApp

### Що змінив і чому

Старий скрипт мав кілька проблем які я виправив:

**1. Все було глобально**
Функції і дані лежали просто у файлі, нічого не було структуровано.
Переписав у клас `ReportApp` — тепер дані і логіка зібрані в одному місці.

**2. Не було валідації**
Оригінальний код рахував всі `paid` замовлення, навіть з від'ємною сумою.
Додав перевірку: враховуються тільки записи де `status = paid` і `amount > 0`.

**3. Не було логування**
Тепер всі події пишуться у `report.log` з часом — запуск, пропущені замовлення, результат запису файлу.

**4. Запис файлу був небезпечний**
Оригінальний `file_put_contents` без прапорців міг зламатись якщо два процеси пишуть одночасно.
Додав `LOCK_EX` — блокує файл на час запису.

---

### Структура класу

| Метод | Що робить |
|---|---|
| `__construct($orders, $file)` | Отримує замовлення і шлях до файлу, логує старт |
| `process()` | Фільтрує замовлення, пропущені записує в лог |
| `summary()` | Рахує кількість, суму і середнє, повертає масив |
| `write()` | Записує результат у файл через `LOCK_EX` |
| `__destruct()` | Викликається автоматично при завершенні, логує кінець |
